# タスクルーティング判定基準 (routing_prompt.md)

このドキュメントは、ユーザー（なべ）の指示を「かえで（Gemini/Antigravity）」が処理すべきか、「しずく（Claude Code）」に依頼すべきかを判定するための基準を定義します。

## 1. 判定の基本方針
- **かえで (Gemini)**: プロジェクト全体の司令塔。UI/UXデザイン、基本の実装、ツールを用いた調査、迅速な修正を得意とする。
- **しずく (Claude)**: コーディングのスペシャリスト。複雑なロジック、大規模なリファクタリング、深い依存関係の解析、一貫性が求められる重厚な実装を得意とする。

## 2. Gemini案件 (かえで) の基準
以下の場合は、かえでがそのまま実行します：
- **UI/UX 関連**: Tailwind CSS を用いたスタイリング、コンポーネントの見た目の調整、アニメーションの追加。
- **基本機能の実装**: 単一ファイルで完結するコンポーネント作成、単純なデータバインド、localStorage への保存処理など。
- **調査・ドキュメント**: 新機能の提案、既存ドキュメントの更新、ライブラリの調査。
- **ボイラープレート作成**: プロジェクトの初期化、ディレクトリ構造の作成。

## 3. Claude案件 (しずく) の基準
以下の場合は、しずくへのパス回し（ルーティング）を提案します：
- **複雑なロジック**: 状態管理 (Pinia) の複雑なアクション設計、データ変換フィルタリングの高度な実装。
- **大規模リファクタリング**: 既存コードの設計変更、複数ファイルにまたがる大規模な構造改修。
- **デバッグ**: 再現困難なバグの追求、パフォーマンス上のボトルネックの解消。
- **一貫性が重要な実装**: 大規模な機能追加において、既存の設計パターンを厳密に継承しつつ整合性を保つ必要がある場合。

## 4. 判定後の対応
- 「しずく」に任せると判断した場合は、なべに対してその理由を簡潔に説明し、`bridge.json` を通じて指示を投げてください。
